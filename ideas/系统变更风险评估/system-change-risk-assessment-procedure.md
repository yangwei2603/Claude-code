# 系统变更风险评估工作程序

> **文件编号**：IT-OP-XXX
> **版本**：V1.0（草案）
> **生效日期**：待审批后生效
> **编制人**：待定
> **审批人**：待定

---

## 1. 目的

为规范系统开发、变更过程中的风险评估工作，确保所有系统层面的变更均经过完整的风险识别、评估与审批流程，防范因系统变更带来的直接风险、衍生风险、流程变更、权限变更及数据变更等后果，特制定本程序。

---

## 2. 适用范围

适用于以下场景：

- **系统新建**：新建业务系统、平台或工具
- **系统变更**：生产环境的功能变更、配置变更、版本升级、架构调整
- **数据变更**：数据结构变更、数据迁移、数据清洗
- **权限变更**：人员账号权限调整、角色变更、特权账号管理
- **接口变更**：系统间接口调整、第三方集成变更
- **下线/退役**：系统下线、数据归档

> **排除范围**：仅涉及个人办公终端配置（如本地软件安装、网络配置调整）且不影响业务系统的变更，经部门经理口头确认后可免于执行本程序。

---

## 3. 职责分工

| 角色 | 职责 |
|------|------|
| **变更发起人** | 填报《系统变更风险评估表》，推动评估与审批流程 |
| **项目经理/系统负责人** | 主导风险识别，组织风险评审，编写风险应对措施 |
| **信息安全负责人**（如有） | 评估信息安全风险、数据安全风险、合规风险 |
| **业务负责人** | 评估业务影响、流程变更风险，确认业务可接受程度 |
| **运维/系统管理员** | 评估技术可行性、基础设施影响、实施复杂度 |
| **审批人** | 根据风险等级按规定级别审批（见第五章） |
| **质量管理/PMO**（如有） | 监督程序执行情况，收集执行数据，推动持续改进 |

## 4. 风险评估维度与内容

### 4.1 直接风险

| 评估项 | 说明 |
|--------|------|
| 系统可用性 | 变更是否导致系统宕机、服务中断？恢复时间预期？ |
| 功能正确性 | 变更是否引入功能缺陷、Bug或逻辑错误？ |
| 数据完整性 | 变更是否可能导致数据丢失、损坏或不一致？ |
| 性能影响 | 变更是否影响系统响应时间、吞吐量、并发能力？ |
| 稳定性 | 变更是否影响系统健壮性、容错能力？ |

### 4.2 衍生风险

| 评估项 | 说明 |
|--------|------|
| 上下游影响 | 变更是否影响依赖此系统的其他系统或业务流程？ |
| 业务连续性 | 是否有业务中断窗口？是否有备用方案？ |
| 回滚风险 | 回滚方案是否可行？回滚时间是否在可接受范围？ |
| 第三方依赖 | 是否依赖第三方服务/组件？其稳定性是否受影响？ |

### 4.3 流程变更风险

| 评估项 | 说明 |
|--------|------|
| 业务流程影响 | 现有业务流程是否需要调整？调整范围多大？ |
| 审批流变化 | 原有关键审批节点是否受影响？ |
| 监控/告警 | 现有监控和告警机制是否需要同步调整？ |

### 4.4 权限变更风险

| 评估项 | 说明 |
|--------|------|
| 账号权限变更 | 新增/修改/删除账号的权限范围是否恰当？ |
| 角色变更 | 角色定义是否需要新增或修改？是否存在权限过大情况？ |
| 特权账号管理 | 特权账号使用是否合规？密码/密钥是否需要更新？ |
| 离职/转岗交接 | 权限回收是否及时？交接清单是否完整？ |

### 4.5 数据变更风险

| 评估项 | 说明 |
|--------|------|
| 数据结构变更 | 表结构、字段变更是否向后兼容？是否需要数据迁移脚本？ |
| 敏感数据处理 | 是否涉及个人隐私、商业秘密等敏感数据？是否需要脱敏？ |
| 数据质量 | 数据迁移后数据质量如何验证？ |
| 数据备份 | 变更前是否完成完整备份？备份可恢复性是否验证？ |

### 4.6 合规风险

| 评估项 | 说明 |
|--------|------|
| 监管合规 | 是否涉及网络安全法、数据安全法、个人信息保护法等要求？ |
| 行业规范 | 是否符合所属行业（金融、医疗、教育等）特定监管要求？ |
| 内部制度 | 是否符合公司信息安全管理制度、数据管理制度等？ |
| 审计追溯 | 变更是否满足审计日志要求？变更记录是否可追溯？ |

## 5. 风险等级判定标准

### 5.1 风险评分矩阵

风险等级由**影响程度**和**发生概率**两个维度综合判定：

**影响程度评分（1-5分）**：

| 分值 | 等级 | 说明 |
|------|------|------|
| 5 | 极高 | 导致核心业务系统长时间中断（>4小时）或重大数据丢失，影响全公司/大量用户 |
| 4 | 高 | 导致核心业务系统短时中断（1-4小时）或重要数据损坏，影响部门核心业务 |
| 3 | 中 | 导致部分功能异常或部分用户受影响（1小时以内），可通过回滚快速恢复 |
| 2 | 低 | 影响范围小，有现成回滚方案，数据基本不受影响 |
| 1 | 极低 | 基本无影响，变更范围可控，可快速验证 |

**发生概率评分（1-5分）**：

| 分值 | 等级 | 说明 |
|------|------|------|
| 5 | 极高 | 在类似变更中经常发生，历史数据高度相关 |
| 4 | 高 | 发生概率较高，缺乏充分测试验证 |
| 3 | 中 | 有一定发生概率，有部分历史案例可参考 |
| 2 | 低 | 发生概率较低，有类似成功案例 |
| 1 | 极低 | 几乎不可能发生，已在其他环境充分验证 |

**风险分值 = 影响程度 × 发生概率**

| 风险分值 | 风险等级 | 颜色标识 |
|----------|----------|----------|
| 1-4 | **低风险** | 绿 |
| 5-9 | **中风险** | 黄 |
| 10-14 | **高风险** | 橙 |
| 15-25 | **极高风险** | 红 |

### 5.2 简化判定规则（快速初判）

对于经验丰富的评估人员，可采用以下简化判定规则直接初判风险等级：

| 变更类型 | 默认风险等级 |
|----------|--------------|
| 配置参数调整（不含网络、安全策略） | 低风险 |
| 非核心功能的增删改，单系统 | 中风险 |
| 核心功能变更，单系统 | 高风险 |
| 数据结构变更（含数据迁移） | 高风险 |
| 跨系统接口/数据交互变更 | 高~极高风险 |
| 系统架构调整、重构 | 极高风险 |
| 权限变更（管理员、特权账号） | 中~高风险 |
| 系统下线/退役 | 高~极高风险 |
| 涉及敏感数据（个人信息、商业秘密） | 高~极高风险 |
| 涉及监管合规（等保、金融监管等） | 极高风险 |

> **注**：简化判定仅供快速初判，最终风险等级以《系统变更风险评估表》评分为准。

## 6. 审批权限矩阵

| 风险等级 | 审批层级 | 适用场景示例 |
|----------|----------|------------|
| **低风险** | 部门经理 | 配置参数调整、文档更新、非核心功能变更 |
| **中风险** | 部门经理 → IT总监/技术负责人 | 一般功能变更、接口调整、权限变更、数据结构变更（无迁移） |
| **高风险** | 部门经理 → IT总监 → CTO/CIO | 核心功能变更、数据迁移、跨系统接口变更、涉及敏感数据 |
| **极高风险** | 部门经理 → IT总监 → CTO/CIO → CEO/分管副总 | 系统架构重构、核心系统下线、涉及监管合规的变更、大规模数据迁移 |

### 审批时效要求

- 低风险：审批人应在收到申请后 **1个工作日** 内完成审批
- 中风险：审批人应在收到申请后 **2个工作日** 内完成审批
- 高风险：审批人应在收到申请后 **3个工作日** 内完成审批，必要时可召开评审会
- 极高风险：审批人应在收到申请后 **5个工作日** 内完成审批，**必须**召开风险评审会

> 评审会参加人员应包括：变更发起人、系统负责人、信息安全负责人、业务负责人、运维负责人、相关审批人。

## 8. 禁止变更的情形

存在以下任一情况的，**一律不得开展变更**，必须先完成整改或制定专项方案并经CEO/分管副总审批：

1. 风险评估未完成或评估表未填写完整的
2. 审批流程未完成或未获得全部必要审批的
3. 未制定回滚方案或回滚方案未经评审的
4. 涉及监管合规要求但未完成合规审查的
5. 核心系统且无备用方案且变更窗口超过容忍窗口的
6. 数据变更未完成备份验证的

## 9. 附则

**培训要求**：本程序发布后30日内，由信息安全负责人组织面向全员的宣贯培训，确保相关人员理解并执行本程序。

**持续改进**：每年度对程序执行数据进行回顾，分析以下指标：
- 风险评估表填写完整率
- 按时审批率
- 变更后问题发生率
- 评审会召开率（高风险及以上变更）

**例外管理**：如因紧急情况（如重大安全漏洞修复）需要立即变更且无法完成完整审批流程的，可由CEO/CTO/CIO口头授权先行实施，事后24小时内补齐全部评估和审批文档。
