# 需求池管理系统 MVP PRD

> 版本：v1.2（评审通过版）
> 日期：2026-05-16
> 状态：评审通过

---

## 1. 背景与问题

### 1.1 现状

当前需求管理使用 Excel/CSV 手动维护，存在以下问题：

| 问题 | 数据支撑 |
|------|---------|
| 状态不透明 | 40.5% 关闭率，59.5% 需求处于活跃状态未流转 |
| 需求评审超期 | 未整合需求已经超过  20 天，仍然没有评审 |
| 描述质量不一 | 35 条描述不足 50 字符，34 条效果描述不足 20 字符 |
| 评审结论没有结构化记录 | 需求无优先级、评审意见、评审结论分类等结构化标注 |
| 下游状态不透明 | 无法了解关联的业务需求/项目当前状态及负责人 |

### 1.2 目标

建立轻量级需求池管理系统，实现：
1. 需求状态透明化，流转可追溯
2. 超期未评审主动预警，避免遗漏
3. 新增待评审需求可视化，聚焦处理
4. 数据结构化，支持统计分析
5. **评审流程规范化，支持标注、驳回、导出**
6. **评审看板可视化，支持 AI 总结**
7. **超期未评审预警：已导入需求超过 20 个工作日仍未评审，说明最后修改时间**
8. **积压超时预警：未关联业务需求且 20 个工作日无操作，说明最后修改时间**
8. **AI 需求质量分析，辅助识别低质量需求**
9. **需求全生命周期可视化，关联项目信息（负责人/状态/计划），预留任务计划扩展**

### 1.3 系统定位

```
数据中心（CSV 导出）→ 需求池（评审、标注、追踪）→ 禅道（结果录入）

┌─────────────────────────────────────────────────────────────────┐
│                         需求池平台（demand）                      │
│  核心：评审 / 分析 / 监控 / 跟踪 原始需求                            │
│                                                                  │
│  输入：数据中心导出的 CSV（原始需求）                                │
│  处理：评审、优先级标注、驳回/通过/未来规划                          │
│  输出：评审结果导出 → 手工录入禅道                                  │
│                                                                  │
│  下游状态（business_status / approval_status）仅作为参考信息展示   │
│  项目信息（负责人/状态/计划）仅作为参考信息展示，不做穿透跳转          │
└─────────────────────────────────────────────────────────────────┘
```

**数据流向**：
1. 从数据中心导出 CSV → 需求池（批量导入）
2. 需求池内进行评审（通过/驳回/未来规划）
3. 评审结果导出 Excel → 手工录入禅道

**不做**：
- 不做 OLAP 上卷下钻分析
- 不做任务层级的穿透追溯
- 不做与禅道的 API 自动集成

## 2. 用户故事

### 2.1 角色定义

| 角色 | 描述 |
|------|------|
| 需求负责人 | 录入需求、更新状态、处理过期 |
| 评审人员 | 对需求进行优先级标注、上线时间排期、驳回/通过 |
| 管理者 | 查看看板、了解整体状况、协调资源 |

### 2.2 核心用户故事

**US-1：需求录入**
> 作为一个需求负责人，我希望能批量导入现有的 CSV 数据，这样我可以快速初始化系统而不用手动录入。

**US-2：状态流转**
> 作为一个需求负责人，我希望能更新需求状态并查看操作记录，这样可以让其他人看到需求进展，也能追溯历史变更。

**US-3：过期预警**
> 作为一个需求负责人，我希望能收到过期提醒，这样我不会遗漏已过期但未处理的需求。

**US-4：积压看板**
> 作为一个管理者，我希望能一眼看到关键指标，这样我可以快速了解需求池健康状况。

**US-5：评审标注**
> 作为一个评审人员，我希望能对每个需求标注优先级、确定上线时间、填写评审意见，这样可以让需求优先级排序有据可依。

**US-6：未评审识别**
> 作为一个评审人员，我希望每次导入 CSV 时系统自动标记未评审的需求，这样我可以聚焦在待评审需求上，不重复评审。

**US-7：版本对比**
> 作为一个评审人员，我希望系统能对比新旧版本自动标记有更新的需求，这样我可以优先评审有变化的需求。

**US-8：驳回处理**
> 作为一个评审人员，我希望能选择驳回类型并填写原因，这样可以让需求提出方清晰知道驳回原因，便于修改后重新提交。

**US-9：评审导出**
> 作为一个评审人员，我希望能导出本次评审的需求清单（需求编号、名称、评审分类、描述、原因等），这样我可以手工录入到禅道系统。

**US-10：评审统计**
> 作为一个管理者，我希望能查看当月评审数据看板，了解评审效率和问题分布，这样我可以持续优化评审流程。

**US-11：AI 总结**
> 作为一个管理者，我希望能配置 AI 大模型自动生成评审摘要，这样可以快速了解评审整体情况并同步给相关干系人。

**US-12：积压超时预警**
> 作为一个管理者，我希望能收到积压预警，告诉我哪些需求超过 20 个工作日无操作且未关联业务需求，这样我可以主动推进而不是遗漏。

**US-13：AI 需求质量分析**
> 作为一个评审人员，我希望能获得每个需求的质量评分和改进建议，这样我可以快速识别哪些需求需要补充说明。

**US-14：项目信息查看**
> 作为一个管理者，我希望能查看每个需求所属项目的负责人、当前状态和计划，这样我可以了解需求的全生命周期，并为后续关联任务计划预留扩展空间。

| 功能 ID | 功能名称 | 描述 | 优先级 |
|---------|---------|------|--------|

| F2 | 单条需求 CRUD | 增删改查单条需求记录 | P0 |
| F3 | 状态流转 | 支持状态变更，记录操作人和时间 | P0 |
| F4 | 过期需求标记 | 自动计算是否过期，区分普经过期和紧急过期 | P0 |
| F5 | 过期预警看板 | 首页突出显示过期紧急需求 | P0 |
| F6 | 评审标注 | 优先级（高/中/低）、评审确定上线时间、评审备注 | P0 |
| F7 | 驳回类型 | 提供标准化驳回类型和原因描述 | P0 |
| F8 | 评审导出 | 导出本次评审需求清单（Excel），字段：需求编号、需求名称、评审分类（通过/驳回/未来规划）、描述、原因。供手工录入禅道使用 | P0 |
| F9 | 积压看板 | 显示总数/活跃/关闭/过期/紧急数量 + 分布图 | P1 |

| F11 | 批量关闭 | 批量选择需求并关闭（需填关闭原因） | P1 |
| F12 | 评审看板 | 当月评审统计（新增/评审数/通过/驳回）、效率指标、驳回原因分布 | P1 |
| F13 | AI 总结 | 调用大模型生成评审摘要，支持飞书/邮件推送 | P2 |
| F14 | 描述质量分布 | 按三字段（demand_desc/businessobjective/businessdesc）长度分布统计 | P1 |
| F15 | 积压超时预警 | 未关联 business_id + 10个工作日无操作 → 预警，显示最后修改时间 | P0 |
| F16 | 未来规划状态 | 评审后暂不推进的需求标记为"未来规划"，单独管理不触发积压预警 | P1 |
| F17 | AI 需求质量分析 | 分析描述长度、完整性、清晰度，输出质量评分和改进建议 | P2 |
| F18 | 下游状态参考 | 需求详情页展示关联的 business_status / approval_status（只读参考） | P1 |
| F19 | 项目信息展示 | 需求详情页展示项目负责人/状态/计划，列表页支持按项目筛选，预留后续关联任务计划扩展 | P1 |

### 3.2 MVP 边界（不做）

- 多租户 / 权限细分
- 与用友 BIP 等外部系统自动集成
- 业务需求/项目管理（仅展示参考信息）
- 移动端适配
- OLAP 上卷下钻分析
- 任务层级的穿透追溯
- 与禅道的 API 自动集成（手工导出 Excel 录入）

| business_name | TEXT | 否 | 关联的业务需求名称 |

### 4.3 状态流转记录表（status_log）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

| to_status | TEXT | 是 | 新 demand_status |
| operator | TEXT | 是 | 操作人 |
| operated_at | TEXT | 是 | 操作时间 |
| remark | TEXT | 否 | 备注（如关闭原因） |

### 4.4 驳回类型定义

| 驳回类型 | 类型编码 | 原因描述 |
|---------|---------|---------|
| 优先级过高 | P001 | 需求紧急程度与业务价值不匹配，建议降低优先级排队 |
| 重复需求 | P002 | 与已有需求重复，功能重叠，建议合并或引用已有需求 |
| 范围不清 | P003 | 需求描述模糊，无法准确评估范围和工作量，需补充说明 |
| 依赖缺失 | P004 | 存在未识别的上游依赖或关联系统，需先解决依赖问题 |
| 资源不足 | P005 | 当前无足够人力或技术资源承接，建议调整时间或拆解 |
| 业务调整 | P006 | 业务方向调整或战略变更，暂不纳入本期范围 |
| 条件不成熟 | P007 | 前提条件不满足（如需先完成XX系统改造），建议暂缓 |
| 其他 | P999 | 其他原因，需在备注中说明 |

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | INTEGER | 是 | 主键，自增 |
| session_date | TEXT | 是 | 评审日期（YYYY-MM-DD） |
| total_count | INTEGER | 是 | 评审需求总数 |
| passed_count | INTEGER | 是 | 通过数 |
| rejected_count | INTEGER | 是 | 驳回数 |
| ai_summary | TEXT | 否 | AI 生成的评审摘要 |
| created_at | TEXT | 是 | 创建时间 |

### 4.6 导入元数据表（import_metadata）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | INTEGER | 是 | 主键，自增 |
| last_import_hash | TEXT | 是 | 上次导入时的全局 hash（用于版本对比） |
| last_import_at | TEXT | 是 | 上次导入时间 |
| total_imported | INTEGER | 是 | 上次导入总数 |

### 4.7 状态定义

| 字段 | 来源 | 说明 |
|------|------|------|
| demand_status | 数据中心，只读 | 原始状态：active / closed |
| plan_status | 系统内部 | 规划状态：正常 / 未来规划 |

**plan_status = 未来规划**：评审后暂不推进的需求，单独管理，不触发积压预警。

**is_closed 标签**：派生字段，当 `demand_status` 为"closed"时 `is_closed=1`。应用层计算，不物理存储。

| is_closed | 含义 | 评审操作 |
|-----------|------|---------|
| 0 | 活跃需求 | 待评审/评审中 |
| 1 | 已关闭需求 | 可查看历史，不可重新评审 |

## 5. 页面与交互

### 5.1 页面结构

```
首页（综合看板）
├── KPI 指标卡（总数/活跃/已关闭/过期/紧急）
├── 过期紧急需求告警区（红色突出）
├── 当月评审统计卡片（本月新增/评审数/通过/驳回）
├── 积压超时预警区（>20个工作日无操作且无关联需求，显示最后修改时间）
├── 描述质量分布（红/黄/绿三档）
├── 状态分布饼图
├── 期望月份分布柱状图
├── 创建人分布柱状图
├── 项目分布饼图/项目状态分布
└── 驳回原因分布（评审看板Tab）

评审管理页
├── 待评审列表（自动高亮未评审+有更新）
├── 已评审列表
├── 评审表单（优先级/上线时间/备注）
├── 驳回弹窗（选择驳回类型+填写原因）
├── 未来规划标记（评审后暂不推进）
└── 导出按钮

需求列表页
├── 搜索栏（按名称/编号搜索）
├── 筛选栏（状态/部门/创建人/评审状态/是否更新/积压/描述质量）
├── 需求表格（可展开查看详情，含质量评分+下游状态参考）
└── 批量操作栏

需求详情页
├── 基本信息展示
├── 项目信息卡片（项目名称、负责人、当前阶段、计划推进时间）
├── 评审信息展示（优先级/上线时间/驳回状态）
├── 下游状态参考（business_status / approval_status / 负责人）
├── 质量评分与改进建议（AI 标签）
├── 状态流转时间线
├── 编辑/关闭/评审按钮
└── 关闭弹窗（需填原因）

导出管理页
├── 导出条件选择（日期范围/评审状态/通过/驳回/未来规划）
├── 导出预览表格（需求编号、需求名称、评审分类、描述、原因）
└── 下载 Excel 按钮

**导出字段说明**：
| 字段 | 来源 | 说明 |
|------|------|------|
| 需求编号 | demand_id | 原始需求编号 |
| 需求名称 | demand_name | 原始需求名称 |
| 优先级 | priority | 评审优先级：高/中/低 |
| 评审分类 | 评审时选择 | 通过 / 驳回 / 未来规划 |
| 上线时间 | review_date | 评审确定上线时间 |
| 描述 | review_notes | 评审备注（通过/未来规划）或驳回原因描述（驳回） |
| 原因 | rejection_type | 驳回类型编码 + 描述（仅驳回时填写） |
```

### 5.2 交互说明

**导入流程**
1. 点击"导入 CSV"按钮
2. 选择本地 CSV 文件
3. 系统解析并显示预览（条数、字段映射）
4. 系统自动对比上一版本，标记：
   - 未评审的需求（高亮）
   - 自上次导入后有字段变化的需求（标记"已更新"）
5. 确认导入，结果提示（导入条数、新增/更新/未评审数量）

**评审流程**
1. 进入"评审管理"页面，系统自动展示：
   - 未评审的需求（优先显示）
   - 已更新但未重新评审的需求
2. 点击需求进入评审表单
3. 填写：优先级（上/中/下）、评审确定上线时间、评审备注
4. 选择：
   - 通过 → 保存评审信息
   - 驳回 → 弹出驳回类型选择 + 原因填写 → 保存
   - 标记为"未来规划" → 保存为 plan_status=未来规划，不触发积压预警
5. 评审完成，系统记录评审人和时间

**导出流程**
1. 进入"导出管理"页面
2. 选择导出条件：日期范围（本月/上个月/自定义）、评审分类（全部/通过/驳回/未来规划）
3. 点击"预览"，显示待导出数据（需求编号、需求名称、评审分类、描述、原因）
4. 点击"导出 Excel"，下载文件
5. 使用导出的 Excel 手工录入禅道

**AI 质量分析**
1. 在需求列表或详情页，显示 AI 质量评分（0-100）
2. 评分 < 60 显示红色警示，60-80 黄色，> 80 绿色
3. 点击评分展开查看：质量改进建议（三段式：业务现状/需求描述/预期效果）
4. 评审人员可参考 AI 建议决定是否驳回

**AI 总结配置**
1. 管理员进入"系统设置"
2. 配置 AI 模型地址（支持本地大模型）
3. 开启/关闭自动总结
4. 配置推送方式（飞书Webhook/邮件）
5. 评审完成后自动触发总结，或手动点击"生成总结"

## 10. 版本对比逻辑

### 10.1 更新检测规则

每次导入 CSV 时，对每条需求进行版本对比：

| 字段 | 检测变更 |
|------|---------|
| demand_name | 是 |
| demand_desc | 是 |
| businessobjective | 是 |
| department | 是 |
| importance | 是 |

**版本 hash 计算**：对上述字段组合计算 MD5，与上次导入的 hash 对比，不一致则标记 `is_updated=1`

### 10.2 导入策略

```
1. 读取 CSV
2. 遍历每条记录：
   a. 如果 demand_id 已存在：
      - 对比版本 hash
      - 有变化则更新数据库 + is_updated=1
      - 保留原有评审信息（评审信息不覆盖）
   b. 如果 demand_id 不存在：
      - 新增记录 + is_reviewed=0
3. 统计：新增数、更新数、未评审数
4. 返回导入报告
```

**注意**：已评审的需求如果 CSV 中有更新，is_updated=1 但 is_reviewed 保持不变，评审状态不重置。评审人员需自行判断是否需要重新评审（通过「已更新」标记筛选）。

## 12. AI 质量分析

### 12.1 评分规则

| 维度 | 计算方式 | 分值权重 |
|------|---------|---------|
| 描述完整性 | demand_desc 长度 >= 50 字符为满分，< 20 字符为 0 分 | 40% |
| 效果清晰度 | businessobjective 长度 >= 20 字符为满分，< 10 字符为 0 分 | 30% |
| 依赖识别 | 检测 demand_desc 中是否提及依赖关系 | 15% |
| 范围明确性 | 检测是否包含具体范围描述（非模糊词） | 15% |

**综合评分**：加权求和，0-100 分

### 12.2 改进建议（三段式）

| 评分区间 | 标签 | 建议类型 |
|---------|------|---------|
| < 40 | 质量差 | "描述过短，建议补充需求背景和具体场景" |
| 40-60 | 质量中 | "效果描述不清晰，建议明确预期成果" |
| 60-80 | 质量良 | "可补充依赖说明，提高评估准确性" |
| > 80 | 质量优 | "需求描述完整，可直接评审" |

**三段式改写输出示例**：
```
业务现状：当前xxx流程依赖人工处理，耗时xxx，容易出错
需求描述：增加xxx功能，实现xxx，支持xxx配置
预期效果：减少人工处理时间xx%，提升数据准确性xx%
```

## 14. 积压超时预警规则

### 14.1 积压超时条件

**积压超时触发条件**（同时满足）：
1. `demand_status = 'active'`
2. `business_id IS NULL`（未关联业务需求）
3. `plan_status IS NULL OR plan_status != '未来规划'`（不是主动暂缓）
4. 最后修改时间距今 > 20 个工作日（周一至周五，排除法定节假日）

**工作日计算规则**：自然日扣除周六周日（周六、周日为非工作日），不单独处理法定节假日调休。

**不触发条件**：
- 已标记为"未来规划"
- 已关联 business_id

### 14.2 积压超时显示

**显示字段**：
| 字段 | 说明 |
|------|------|
| demand_id | 需求编号 |
| demand_name | 需求名称 |
| created_date | 创建时间 |
| last_modified | 最后修改时间 |
| days_since_modified | 距今天数 |
| creator | 创建人 |

### 14.3 预警显示

首页"积压超时预警区"，显示超过 20 个工作日无操作的需求列表，按最后修改时间倒序排列，超期天数标红。
