# 需求池管理系统 v2 PRD

> 版本：v2.0（✅ 已验收）

---

## 1. 背景与目标

### 1.1 现状问题

v1 系统（requirement-pool-hermes）解决了基础的评审流程，但存在以下问题：

| # | 问题 | 影响 |
|---|------|------|
| 1 | 导航栏顺序混乱 | 用户找不到对应功能入口 |
| 2 | 首页看板数据无法下钻到明细 | 只能看数字，无法追溯具体需求 |
| 3 | 看板模块拼凑，两套UI不统一 | 需要整合两套参考项目的优势 |
| 4 | CSV 导入缺少明确模板 | 用户不知道如何准备导入数据 |
| 5 | 评审管理 UI/功能分离 | 两套参考各有利弊，需取长补短 |
| 6 | 评审通过后项目归属不明确 | 需求与项目脱节，全生命周期断裂 |
| 7 | 评审结果无持久化追溯 | 二次导入时无法识别已评审需求 |
| 8 | 评审导出功能冗余 | 导出功能应合并到评审管理模块 |
| 9 | 缺少需求全生命周期管理 | 评审通过后的需求无状态跟踪 |
| 10 | 筛选维度不足 | 大数据量下无法有效筛选 |
| 11 | 缺少智能问答入口 | 无法直接对数据库进行多维度分析 |

### 1.2 目标

在 v1 基础上，完成以下改进：

1. **整合首页看板** — 统一 UI，保留两套参考的优势指标，增加下钻能力
2. **完善评审流程** — 评审结果持久化，支持版本对比，支持项目归属
3. **新增需求全生命周期管理** — 评审通过 → 立项 → 上线 → 关闭
4. **增强筛选能力** — 关键维度全部暴露为筛选器
5. **集成智能问答** — 一键启动 Claude CLI 进入数据分析工作空间

## 2. 系统架构

### 2.1 数据流向

```
数据中心 CSV 导出
      ↓
需求池 v2（批量导入）
      ↓
┌─────────────────────────────────────────────────────────────┐
│  评审管理                                                    │
│    ├─ 待评审需求（未评审 + 已评审未整合 + 有更新）             │
│    ├─ 本次评审（本次新评）                                   │
│    └─ 历史评审（已录入禅道）                                  │
└─────────────────────────────────────────────────────────────┘
      ↓（评审通过后）
┌─────────────────────────────────────────────────────────────┐
│  需求全生命周期                                              │
│    ├─ 已通过（待立项）                                        │
│    ├─ 已立项（进行中）                                        │
│    ├─ 已上线                                                  │
│    └─ 已关闭                                                  │
│                                                         │
│  项目管理（评审通过无归属需求 → 独立展示 → 提醒立项）        │
└─────────────────────────────────────────────────────────────┘
      ↓（手工录入）
禅道
```

### 2.2 导航结构

按以下顺序组织：

1. **首页看板** — 核心指标 + 多维度分布 + 积压超时预警
2. **CSV 导入** — 批量导入 + 导入模板下载
3. **评审管理** — 待评审 + 已评审 + 历史记录
4. **需求全生命周期** — 按状态分类（已通过 / 已立项 / 已上线 / 已关闭）

## 3. 功能详情

### 3.1 首页看板

**保留模块（来自 requirement-pool-hermes）：**

- 指标卡：总需求数、进行中、已关闭、待评审、本月新增
- 多维度描述质量分布（饼图 + 评分分布）

**新增/替换模块（来自 requirement-pool UI）：**

- 积压超时预警：超过 20 个工作日未评审的需求列表，支持下钻到详情
- 当月评审统计：本月评审通过/驳回/未来规划数量
- 状态分布：各需求状态的数量和占比
- 创建人分布：按创建人聚合的需求数量
- 创建部门/科室分布：按部门聚合的需求数量
- 进行中的项目：已立项项目的需求数量和负责人

**下钻能力：**

所有分布图点击后跳转至对应的筛选视图（如点击某创建人 → 该创建人的全部需求列表）。

**无归属项目需求提醒（来自需求全生命周期）：**

首页新增提示区：展示「评审通过但未关联项目」的需求列表，提醒项目经理尽快立项。

---

### 3.2 CSV 导入

**功能实现（按 requirement-pool-hermes）：**

- 批量导入需求（支持 CSV 文件）
- 导入时自动对比历史评审记录，标记：
  - 已评审需求（标记为「已评审」）
  - 已评审但有更新（标记为「已评审未整合」，重新进入待评审）
  - 新需求（标记为「待评审」）
- 导入结果展示：成功数量 + 各状态分布

**导入模板：**

提供标准 CSV 模板下载，包含所有必填字段和示例行。

---

### 3.3 评审管理

**UI 参考 requirement-pool-hermes，功能按 requirement-pool：**

- 待评审需求列表（支持筛选：创建人、部门、创建时间范围、优先级）
- 已评审需求列表（本次评审）
- 历史评审记录（已确认录入禅道）

**评审信息录入：**

| 字段 | 说明 |
|------|------|
| 需求编号 | 自动带入，不可修改 |
| 需求名称 | 自动带入，不可修改 |
| 评审分类 | 通过 / 驳回 / 未来规划（必选） |
| 优先级 | 高 / 中 / 低 / 未标注（通过时必选） |
| 评审意见 | 多行文本 |
| 驳回类型 | 描述不清 / 重复需求 / 不属于规划 / 其他（驳回时必选） |
| 驳回原因 | 多行文本（驳回时必填） |
| 计划上线时间 | 日期（通过时必选） |

**通过时的项目归属：**

- 选择「新建项目」→ 输入项目名称 + 负责人
- 选择「归属现有项目」→ 从项目列表下拉选择

**本次评审结果的二次确认：**

录入禅道后，点击「确认录入禅道」按钮，该条记录从「本次评审」移入「历史评审」，并记录确认时间。

---

### 3.4 评审结果追溯

**评审结果追溯 storage_design：**

`requirement_review` 表以 `demand_id` 为主键，每次评审更新记录，不创建新行。

| 字段 | 类型 | 说明 |
|------|------|------|
| demand_id | String (PK) | 需求编号 |
| priority | String | 优先级（高/中/低/未标注） |
| review_date | String | 评审日期 |
| review_notes | String | 评审意见 |
| rejection_type | String | 驳回类型（描述不清/重复需求/不属于规划/其他） |
| rejection_reason | String | 驳回原因 |
| reviewer | String | 评审人 |
| reviewed_at | String | 评审时间 |
| version_hash | String | 数据版本哈希（MD5，用于判断是否有更新） |
| is_reviewed | Integer | 是否已评审（0/1） |
| is_updated | Integer | 是否有更新（0/1，hash 变化时置 1） |
| plan_status | String | 规划状态（通过/驳回/未来规划） |
| project_id | String | 关联项目ID（通过时填写） |
| is_confirmed | Integer | 是否已确认录入禅道（0/1） |
| confirmed_at | String | 确认录入禅道时间 |

**版本对比逻辑：**

每次导入时，根据 CSV 中 `demand_id` 查找已有评审记录：
- 无历史记录 → 标记为「待评审」，新建评审记录（`is_reviewed=0`）
- 有历史记录且 hash 未变化 → 保持现有评审状态
- 有历史记录但 hash 变化 → 标记 `is_updated=1`，`is_reviewed` 重置为 0，重新进入待评审

### 3.5 需求全生命周期管理

**状态流转：**

```
待评审 → 已评审（通过）→ 已立项 → 已上线 → 已关闭
                    ↓
              驳回（进入评审管理的已评审列表）
                    ↓
              未来规划（进入评审管理的已评审列表）
```

**各状态功能：**

| 状态 | 功能 |
|------|------|
| 已通过 | 展示需求信息 + 关联项目（待立项） |
| 已立项 | 展示需求信息 + 项目信息（负责人/计划上线时间） |
| 已上线 | 标记上线日期 |
| 已关闭 | 标记关闭日期 + 关闭原因 |

**无归属项目需求：**

- 独立展示区（非挂靠在任何项目下的通过需求）
- 首页看板提醒区域展示
- 支持快速创建项目并关联

## 8. CSV 导入模板

**模板下载：** 系统提供 `demand_import_template.csv` 下载，包含以下字段：

| 字段名 | 必填 | 格式说明 | 示例 |
|--------|------|---------|------|
| demand_id | ✅ | 字符串，唯一标识 | REQ-2024-0001 |
| demand_name | ✅ | 字符串 | 用户权限优化 |
| demand_desc | | 字符串 | 当前权限配置过于复杂 |
| businessobjective | | 字符串 | 提升用户体验 |
| department | | 字符串 | 数字化办公室 |
| creator | | 字符串 | 张三 |
| business_created_date | | YYYY-MM-DD | 2024-01-15 |
| deadline | | YYYY-MM-DD | 2024-06-30 |
| importance | | 高/中/低 | 高 |
| reason_type | | 字符串 | 用户体验 |
| source_dept | | 字符串 | 营销部 |
| business_capability | | 字符串 | 用户运营 |
| project_id | | 字符串，来源于导入数据 | PROJ-001 |
| project_name | | 字符串 | 用户中心升级 |
| business_status | | 字符串 | 实施中 |
| approval_status | | 字符串 | 待审批 |
| last_modified | | YYYY-MM-DD HH:mm:ss | 2024-05-01 10:30:00 |

**version_hash 计算规则：** 对 `demand_desc + businessobjective + department + importance + reason_type` 拼接后取 MD5，作为 `version_hash` 字段值。导入时与库中已有 hash 对比判断是否有更新。

## 9. 积压超时预警规则

- 定义：需求导入后超过 **20 个工作日** 未完成评审，触发预警
- 计算方式：排除周末（周六、周日），从 `business_created_date` 起算
- 展示：首页看板「积压超时预警」区域，显示超时需求列表（demand_id、demand_name、超期天数、最后修改时间）
- 下钻：点击需求跳转至评审管理页面，自动带上该需求的筛选条件
